## 3.1 类加载器

HotSpot VM 中的类加载器体系遵循经典的**双亲委派模型**，主要由三类加载器构成：

1.  **引导类加载器 (Bootstrap ClassLoader)**：由 C++ 实现，负责加载核心类库（如 `rt.jar`）。
2.  **扩展类加载器 (ExtClassLoader)**：由 Java 实现，负责加载 `JAVA_HOME/lib/ext` 目录下的扩展类库。
3.  **应用类加载器 (AppClassLoader)**：由 Java 实现，负责加载应用程序 `classpath` 下的类。

这三者的关系是：AppClassLoader 的父加载器是 ExtClassLoader，而 ExtClassLoader 的父加载器是 Bootstrap ClassLoader（通过 `parent` 字段为 `null` 表示）。

---
### 1. 引导类加载器 (Bootstrap ClassLoader)

**核心代码位置**：`openjdk/hotspot/src/share/vm/classfile/classLoader.cpp`

```cpp
instanceKlassHandle ClassLoader::load_classfile(Symbol* h_name, TRAPS) {
  // ... 获取文件名称 ...
  stringStream st;
  st.print_raw(h_name->as_utf8());
  st.print_raw(".class");
  char* name = st.as_string();

  // 根据文件名查找 Class 文件
  ClassFileStream* stream = NULL;
  int classpath_index = 0;
  {
    ClassPathEntry* e = _first_entry;
    while (e != NULL) {
      stream = e->open_stream(name, CHECK_NULL);
      if (stream != NULL) {
        break;
      }
      e = e->next();
      ++classpath_index;
    }
  }

  instanceKlassHandle h;
  if (stream != NULL) {
    ClassFileParser parser(stream);
    ClassLoaderData* loader_data = ClassLoaderData::the_null_class_loader_data();
    Handle protection_domain;
    TempNewSymbol parsed_name = NULL;
    // 加载并解析 Class 文件，注意此时并未开始连接
    instanceKlassHandle result = parser.parseClassFile(h_name,
                                                       loader_data,
                                                       protection_domain,
                                                       parsed_name,
                                                       false,
                                                       CHECK);
    // 调用 add_package() 函数，把当前类的包名加入 _package_hash_table 中
    if (add_package(name, classpath_index, THREAD)) {
      h = result;
    }
  }
  return h;
}
```

**详细解读**：

*   **功能定位**：这是引导类加载器的核心实现，负责从 `bootclasspath` 中查找并加载指定的类文件。
*   **查找逻辑**：它通过遍历一个由 `ClassPathEntry` 构成的单链表，逐个目录或 Jar 包查找目标类。`open_stream()` 方法是一个虚函数，不同的 `ClassPathEntry` 实现（如目录、Jar包）有不同的打开方式。
*   **解析过程**：找到类文件后，通过 `ClassFileParser` 解析字节码，生成 `instanceKlass` 对象。注意，这里的解析只是将字节码转换为 HotSpot 内部的 C++ 对象（如 `InstanceKlass` 和 `ConstantPool`），并未完成类的连接（验证、准备、解析）阶段。
*   **包缓存**：调用 `add_package()` 将类的包名缓存起来，用于后续的包访问权限检查，避免重复解析。

**原理**：引导类加载器是 JVM 启动时最早初始化的加载器，它加载了 `java.lang.Object`、`java.lang.ClassLoader` 等最基础的类，是整个 Java 世界的基石。

### 2. 扩展类加载器 (ExtClassLoader)

**核心代码位置**：`openjdk/jdk/src/share/classes/sun/misc/Launcher.java`

```java
static class ExtClassLoader extends URLClassLoader {
    // 构造函数
    public ExtClassLoader(File[] dirs) throws IOException {
        super(getExtURLs(dirs), null, factory);
    }

    public static ExtClassLoader getExtClassLoader() throws IOException {
        final File[] dirs = getExtDirs();
        return new ExtClassLoader(dirs);
    }

    private static File[] getExtDirs() {
        String s = System.getProperty("java.ext.dirs");
        File[] dirs;
        if (s != null) {
            StringTokenizer st = new StringTokenizer(s, File.pathSeparator);
            int count = st.countTokens();
            dirs = new File[count];
            for (int i = 0; i < count; i++) {
                dirs[i] = new File(st.nextToken());
            }
        } else {
            dirs = new File[0];
        }
        return dirs;
    }
}
```

**详细解读**：

*   **继承关系**：`ExtClassLoader` 继承自 `URLClassLoader`，这是一个通用的、可以从 URL（包括本地文件和网络资源）加载类的加载器。
*   **父加载器**：在构造函数中，调用父类构造器时，第二个参数 `parent` 被设置为 `null`。这表示它的父加载器是引导类加载器。
*   **加载路径**：`getExtDirs()` 方法从系统属性 `java.ext.dirs` 中读取扩展类库的路径，通常是 `JAVA_HOME/lib/ext`。

**原理**：扩展类加载器的设计是为了让开发者可以方便地扩展 JDK 的功能，而无需修改核心类库。它遵循双亲委派，先委托给父加载器（引导类加载器）加载，失败后再自己尝试加载。

### 3. 应用类加载器 (AppClassLoader)

**核心代码位置**：`openjdk/jdk/src/share/classes/sun/misc/Launcher.java`

```java
static class AppClassLoader extends URLClassLoader {
    AppClassLoader(URL[] urls, ClassLoader parent) {
        super(urls, parent, factory);
    }

    public static ClassLoader getAppClassLoader(final ClassLoader extcl) throws IOException {
        final String s = System.getProperty("java.class.path");
        final File[] path = (s == null) ? new File[0] : getClassPath(s);
        return new AppClassLoader(urls, extcl);
    }

    public Class loadClass(String name, boolean resolve) throws ClassNotFoundException {
        return (super.loadClass(name, resolve));
    }
}
```

**详细解读**：

*   **父加载器**：在构造函数中，`parent` 参数被设置为 `ExtClassLoader` 的实例。这明确了双亲委派关系：AppClassLoader -> ExtClassLoader -> Bootstrap ClassLoader。
*   **加载路径**：`getAppClassLoader()` 方法从系统属性 `java.class.path` 中读取应用程序的类路径，这是我们最常打交道的加载路径。
*   **加载逻辑**：`loadClass()` 方法直接调用了父类 `URLClassLoader` 的实现，而 `URLClassLoader` 的 `loadClass()` 又遵循了 `ClassLoader` 定义的双亲委派模型。

**原理**：应用类加载器是我们日常开发中最常用的加载器，它负责加载我们自己编写的代码和第三方依赖库。当我们执行 `java Main.class` 时，`Main` 类就是由它加载的。

### 4. 加载器实例的创建与初始化

**核心代码位置**：`sun.misc.Launcher` 和 `java.lang.ClassLoader`

```java
// 在 Launcher 类中
private static Launcher launcher = new Launcher();

public Launcher() {
    // 1. 首先创建扩展类加载器
    extcl = ExtClassLoader.getExtClassLoader();
    // 2. 以 ExtClassLoader 为父加载器创建 AppClassLoader
    loader = AppClassLoader.getAppClassLoader(extcl);
    // 3. 设置当前线程的上下文类加载器为 AppClassLoader
    Thread.currentThread().setContextClassLoader(loader);
}

// 在 ClassLoader 类中
private static ClassLoader scl;

public static ClassLoader getSystemClassLoader() {
    initSystemClassLoader();
    return scl;
}

private static synchronized void initSystemClassLoader() {
    if (!sclSet) {
        sun.misc.Launcher l = sun.misc.Launcher.getLauncher();
        if (l != null) {
            scl = l.getClassLoader(); // 返回的就是 AppClassLoader
        }
        sclSet = true;
    }
}
```

**详细解读**：

*   **Launcher 初始化**：JVM 启动时，会创建一个 `Launcher` 单例对象。它的构造函数按顺序创建了 ExtClassLoader 和 AppClassLoader。
*   **系统类加载器**：`ClassLoader.getSystemClassLoader()` 是一个公开的 API，它返回的 `scl` 变量，正是 `Launcher` 中创建的 `AppClassLoader` 实例。这就是为什么我们说“系统类加载器”就是应用类加载器。
*   **上下文类加载器**：`Launcher` 构造函数的最后一步，将当前线程（即主线程）的上下文类加载器设置为 `AppClassLoader`。这是为了解决双亲委派模型在某些场景下的局限性，例如 SPI（Service Provider Interface）框架。

### 5. HotSpot VM 与 Java 层的交互

**核心代码位置**：`hotspot/src/share/vm/classfile/systemDictionary.cpp`

```cpp
void SystemDictionary::compute_java_system_loader(TRAPS) {
  // 调用 Java 层的静态方法 java.lang.ClassLoader.getSystemClassLoader()
  JavaValue result(T_OBJECT);
  JavaCalls::call_static(&result,
                         vmSymbols::java_lang_ClassLoader(),
                         vmSymbols::getSystemClassLoader_name(),
                         vmSymbols::void_classloader_signature(),
                         CHECK);

  // 将返回的 AppClassLoader 对象引用保存到 _java_system_loader
  _java_system_loader = (oop)result.l();
}
```

**详细解读**：

*   **跨层调用**：HotSpot VM 是用 C++ 写的，但它需要知道 Java 层创建的系统类加载器（AppClassLoader）是谁。`compute_java_system_loader()` 函数通过 `JavaCalls::call_static()` 这个关键 API，回调了 Java 层的 `ClassLoader.getSystemClassLoader()` 方法。
*   **结果保存**：调用返回的结果是一个指向 `AppClassLoader` 对象的 OOP（Ordinary Object Pointer），VM 将其保存在 `_java_system_loader` 全局变量中，后续加载主类等操作就会使用这个加载器。

**原理**：这完美体现了 JVM 的分层架构。底层的 C++ 虚拟机通过精心设计的接口，与上层的 Java 运行时环境进行交互，共同完成复杂的类加载任务。

## 三、总结：双亲委派模型的完整链路

1.  **用户请求**：应用代码调用 `classLoader.loadClass("com.example.MyClass")`。
2.  **委派开始**：`AppClassLoader` 首先检查自己是否已加载过此类，若没有，则将请求委派给父加载器 `ExtClassLoader`。
3.  **再次委派**：`ExtClassLoader` 同样检查缓存，未命中则委派给它的父加载器，即 `Bootstrap ClassLoader`。
4.  **顶层加载**：`Bootstrap ClassLoader` 在 `bootclasspath` 中查找并尝试加载。如果找到，返回 `Class` 对象；如果找不到，抛出异常。
5.  **子加载器尝试**：如果顶层加载失败，异常会逐层向下传递。`ExtClassLoader` 收到后，会在 `java.ext.dirs` 路径下尝试加载。
6.  **最终加载**：如果 `ExtClassLoader` 也加载失败，请求最终回到 `AppClassLoader`，它会在应用 `classpath` 中查找并加载类。如果所有加载器都失败，才会抛出 `ClassNotFoundException`。
